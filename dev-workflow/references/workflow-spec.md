# 开发流程规范 v1.0

> 基于 T-20260409 系列任务实践抽象

---

## 一、任务分级

| Level | 链路 | 场景 |
|-------|------|------|
| **L0** | 协调者 → 用户 | 纯问答，无需产物 |
| **L1** | 规划者 → 用户 | 需要规划但无实际产物 |
| **L2** | 规划者 → 执行者 → 用户 | 需要实际产物 |
| **L3** | 规划者 → 执行者 → 审查者 → 用户 | 高风险或需要独立审查 |

**默认规则**：
- 涉及代码/配置修改 → **L2**
- 安全相关/高风险/关键功能 → **L3**
- 不确定级别 → 默认进入 handoff

---

## 二、角色定义

| 角色 | 职责 | 边界 |
|------|------|------|
| **协调者** | 接收请求、任务分级、状态管理、结果交付 | 不做具体规划、执行、审查 |
| **规划者** | 理解目标、制定计划、识别风险、定义验收标准 | 不做代码实现、文件操作 |
| **执行者** | 代码实现、文件操作、测试验证、工具调用 | 不做规划决策、独立审查 |
| **审查者** | 独立验证、问题发现、质量评估、结果对比 | 不做代码修改 |

### 角色能力矩阵

| 能力 | 协调者 | 规划者 | 执行者 | 审查者 |
|------|--------|--------|--------|--------|
| 理解用户请求 | ✅ | ✅ | ❌ | ❌ |
| 任务分级 | ✅ | ❌ | ❌ | ❌ |
| 制定计划 | ❌ | ✅ | ❌ | ❌ |
| 代码实现 | ❌ | ❌ | ✅ | ❌ |
| 文件操作 | ❌ | ❌ | ✅ | ❌ |
| 调用外部工具 | ❌ | ❌ | ✅ | ✅ |
| 独立验证 | ❌ | ❌ | ❌ | ✅ |
| 状态管理 | ✅ | ❌ | ❌ | ❌ |
| 结果交付 | ✅ | ❌ | ❌ | ❌ |

## 三、工具组合

### 工具类型

| 类型 | 功能 | 调用时机 |
|------|------|----------|
| **实现工具** | 代码编写、修改、重构 | 执行阶段 |
| **分析工具** | 代码扫描、质量评估、问题检测 | 执行/审查阶段 |
| **审查工具** | 代码审查、安全检查、最佳实践验证 | 执行/审查阶段 |
| **测试工具** | 单元测试、集成测试、端到端测试 | 执行阶段 |

> **说明**：本规范不绑定特定工具，具体工具由用户配置决定。
> 支持 CLI 工具（如 CodeBuddy、Codex、Claude Code 等）、subagent、自定义脚本等。

### 组合策略

```yaml
# 标准任务 (L2)
execution:
  primary: 用户配置的实现工具
  secondary: 用户配置的审查工具 (可选)

# 严格任务 (L3)
execution:
  primary: 用户配置的实现工具
  secondary: 用户配置的审查工具 (必须)
review:
  primary: 用户配置的审查工具 或 分析工具

# 双重检查机制
round_1: 工具A (全面检查)
round_2: 工具B (独立验证)
final: 对比分析 → 综合结论
```

## 四、流程阶段

### Intake（接收）

```yaml
输入: 用户原始请求
处理:
  - 标准化请求描述
  - 任务分级判断
  - 创建任务 ID
输出: Intake 文件
```

### Planning（规划）

```yaml
输入: Intake 文件
处理:
  - 分析目标
  - 制定执行步骤
  - 识别风险
  - 定义验收标准
输出: Plan 文件（含验收标准）
```

### Confirmation（确认）

```yaml
输入: Plan 文件
处理:
  - 向用户展示计划概要
  - 等待用户确认
输出: Confirmation 文件
决策:
  - 确认 → 进入执行
  - 拒绝 → 回退规划
  - 取消 → 终止任务
```

### Building（执行）

```yaml
输入: Plan 文件 + Confirmation 文件
处理:
  - 调用实现工具
  - 执行代码修改/文件操作
  - 调用审查工具（如需要）
  - 验证执行结果
输出: Build 文件（含工具调用证据）
```

### Reviewing（审查）

```yaml
输入: Build 文件
处理:
  - 独立验证执行结果
  - 发现遗漏问题
  - 评估质量
  - 给出结论
输出: Review 文件（含审查结论）
```

### Delivery（交付）

```yaml
输入: 各阶段文件
处理:
  - 汇总结果
  - 生成最终报告
输出: Delivery 文件 → 用户
```

## 五、文件契约

### 命名规范

```
{T-任务ID}/
├── 00-intake.yaml       # 接收
├── 10-plan.yaml         # 规划
├── 15-confirmation.yaml # 确认
├── 20-build.yaml        # 执行
├── 30-review.yaml       # 审查（L3）
└── 40-delivery.yaml     # 交付
```

### 状态定义

```yaml
states:
  - created     # 已创建
  - completed   # 已完成
  - confirmed   # 已确认
  - rejected    # 已拒绝
  - blocked     # 已阻塞
  - partial     # 部分完成
```

### 关键内容模板

**Intake**：
```yaml
task_id: 任务ID
level: L0/L1/L2/L3
user_request: 原始请求
normalized_request: 标准化描述
```

**Plan**：
```yaml
objective: 目标
scope: 范围
plan: 步骤列表
acceptance_criteria: 验收标准
risks: 风险识别
```

**Build**：
```yaml
artifacts_modified: 修改文件
tool_calls: 工具调用记录
verification: 验证结果
against_criteria: 验收对照
```

**Review**：
```yaml
verdict: 结论（approved/needs_revision/rejected）
findings: 发现的问题
recommendations: 改进建议
```

## 六、质量保证

### 证据链

```yaml
L2 任务:
  - 规划证据: Plan 文件
  - 执行证据: Build 文件 + 工具调用记录
  - 验收证据: 对照验收标准

L3 任务:
  - 规划证据: Plan 文件
  - 执行证据: Build 文件 + 双重工具调用记录
  - 审查证据: Review 文件
  - 验收证据: 对照验收标准 + 审查结论
```

### 验收检查

```yaml
before_delivery:
  - 检查工具调用记录是否存在
  - 检查修改文件与计划是否一致
  - 检查验收标准是否全部满足
  - 检查审查结论（L3）是否通过
```

---

## 七、异常处理

```yaml
超时恢复:
  1. 记录部分完成状态
  2. 重新派发恢复任务
  3. 检查当前进度
  4. 继续未完成工作
  5. 输出最终结果

阻塞处理:
  1. 记录阻塞原因
  2. 通知用户
  3. 等待外部输入或决策
```

## 八、最佳实践

### 任务设计原则

- 单一目标、明确边界、可验收标准
- 避免：跨多模块的大任务、目标模糊的需求、无法验证的工作

### 规划原则

- 步骤清晰、风险识别充分、验收标准明确、范围边界清晰
- 避免：步骤模糊、遗漏风险、无法验收

### 执行原则

- 留下工具调用证据、验证每个步骤结果、对照验收标准检查、记录偏离和原因
- 避免：无证据输出、跳过验证、偏离计划未记录

### 审查原则

- 独立验证、发现遗漏和偏差、给出明确结论、提供改进建议
- 避免：流于形式、遗漏关键问题、结论模糊
