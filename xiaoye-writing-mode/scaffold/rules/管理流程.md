# 管理流程

## 概述

Coordinator模块的工作流程规范，定义项目管理的标准操作流程。

## 项目生命周期

### 1. 项目创建

```
触发条件：用户开始新小说

操作步骤：
1. 使用 novel-cli novel create 创建小说
2. 在 memory/projects/ 下创建项目管理目录
3. 初始化 todo.md 和 progress.md
4. 更新 MEMORY.md 项目列表
5. 更新 novels-archive/index.md
```

### 2. 项目进行中

```
日常操作：
- 创作任务分配和追踪
- 评审任务分配和追踪
- TODO更新和优先级调整
- 进度数据更新
- 质量趋势监控
```

### 3. 项目暂停/恢复

```
暂停操作：
1. 更新项目状态为"暂停"
2. 记录暂停原因
3. 整理当前TODO清单
4. 生成阶段性总结

恢复操作：
1. 更新项目状态为"进行中"
2. 检查TODO清单，调整优先级
3. 回顾上次进度和遗留问题
```

### 4. 项目完成

```
完成操作：
1. 更新项目状态为"完成"
2. 生成项目总结报告
3. 整理归档所有文档
4. 提取经验教训到MEMORY.md
```

## 任务管理流程

### TODO创建规则

```markdown
- [ ] 任务描述 | 优先级:🔴🟡🟢 | 模块:Writer/Reviewer | 创建日期:YYYY-MM-DD
```

### TODO状态流转

```
新建 → 进行中 → 已完成
  ↓         ↓
取消     阻塞（需外部输入）
```

### 优先级定义

| 级别 | 含义 | 响应要求 |
|------|------|---------|
| 🔴 高 | 影响当前创作进度 | 优先处理 |
| 🟡 中 | 影响质量但不阻塞 | 计划内处理 |
| 🟢 低 | 优化改进类 | 有空时处理 |

## 创作评审协调流程

### 标准流程：创作 → 评审 → 迭代

```
用户指令："创作并评审第三章"

Step 1: Coordinator创建任务
  - 创建写作任务 → Writer
  - 预创建评审任务 → Reviewer（待创作完成后激活）

Step 2: Writer执行创作
  - ⭐ 前置条件检查：读取项目设定 + 前3章章节简介
  - 读取项目设定和前文
  - 应用辩证写作方法
  - 产出创作内容
  - 通知Coordinator完成

Step 3: Coordinator归档 + ⭐更新章节简介
  - 归档创作内容
  - [MODIFY_TRIGGER] 更新章节简介
  - 转交Reviewer

Step 4: Reviewer执行评审
  - 应用评审标准体系
  - 产出评审报告
  - 通知Coordinator完成

Step 5: Coordinator整合结果
  - 更新进度和TODO
  - ⭐ 如章节被修改，更新章节简介
  - 向用户展示创作+评审结果
  - 记录到项目档案
```

### 快速流程：仅创作

```
用户指令："直接写第三章"

Step 1: Writer执行创作（内置快速辩证自检）
  - ⭐ 前置条件检查：读取前3章章节简介
Step 2: Coordinator更新进度 + ⭐更新章节简介
Step 3: 向用户展示结果
```

### 迭代流程：根据评审修改

```
用户指令："根据评审结果修改第一章"

Step 1: Coordinator调取评审报告
Step 2: Writer根据评审建议修改
Step 3: Reviewer复评修改后内容
Step 4: Coordinator更新记录 + ⭐[MODIFY_TRIGGER]更新章节简介
Step 5: 向用户展示修改结果
```

## 进度追踪规范

### 每次创作后更新

```markdown
## [项目名] 进度更新
- 日期：YYYY-MM-DD
- 操作：创作/评审/修改
- 章节：第X章
- 字数：XXXX字（新增/修改）
- 评审分数：X.X⭐（如有）
```

### 定期汇总

每完成一个重要节点（如完成一章、完成一个故事弧），更新progress.md：

```markdown
## [项目名] 阶段性进度
- 截止日期：YYYY-MM-DD
- 完成章节：X章
- 总字数：XXXXX字
- 平均评审分：X.X⭐
- 质量趋势：提升/稳定/下降
- 待办事项数：X项
- 下阶段计划：[简述]
```

## 数据归档规范

### 评审记录归档

```
评审完成后：
1. 评审报告存入 memory/projects/[项目名]/reviews/
2. 文件命名：章节X-评审报告.md
3. 更新 progress.md 的评审数据
4. ⭐ 如章节被修改，更新章节简介
```

### 创作内容归档

```
创作完成后：
1. 章节内容存入 novels-archive/[项目名]/章节/
2. 设定文档存入 novels-archive/[项目名]/设定/
3. 更新 novels-archive/[项目名]/项目索引.md
4. ⭐ [MODIFY_TRIGGER] 更新章节简介（强制触发，不可跳过）
```

### ⭐ 章节简介归档规范（新增）

```
章节简介管理规则：

存储位置：novels-archive/[项目名]/章节/章节简介/第{n}章-{标题}.md

触发时机：
1. 创作新章节后 → 自动生成对应章节简介文件
2. 修改章节内容后 → [MODIFY_TRIGGER] 强制更新对应章节简介文件
3. 评审后修改章节 → [MODIFY_TRIGGER] 强制更新对应章节简介文件

简介格式（每章独立文件）：
# 第{n}章：{标题}

## 内容概述
[100-200字概述本章核心事件]

## 情节发展
- [关键事件和转折点]

## 角色成长
[主要角色的状态变化]

## 衔接点
- [承上启下的关键信息]

## 伏笔线索
- [遗留的伏笔和待解决的线索]

## 最后更新
YYYY-MM-DD

强制触发机制 [MODIFY_TRIGGER]：
- 任何章节修改操作必须触发管理模块
- 管理模块必须更新章节简介
- 不可跳过，不可延迟
```

## 状态报告规范

### 按需生成

用户请求时生成项目状态报告，包含：
- 项目基本信息
- 整体进度概览
- 质量趋势分析
- 当前TODO清单
- 风险提示
- 改进建议

### 报告模板

```markdown
# 项目状态报告

生成时间：YYYY-MM-DD HH:MM

## 项目基本信息
- 名称：[项目名]
- 后端ID：[novelId]
- 状态：进行中/暂停/完成
- 开始日期：YYYY-MM-DD

## 进度概览
- 计划章节：XX章
- 已完成：X章
- 完成率：XX%
- 总字数：XXXXX字

## 质量趋势
- 平均评审分：X.X⭐
- 最高分：X.X⭐（第X章）
- 最低分：X.X⭐（第X章）
- 趋势：↑提升 / →稳定 / ↓下降

## 待办事项
- 🔴 高优先级：X项
- 🟡 中优先级：X项
- 🟢 低优先级：X项

## 风险提示
- [如有]

## 改进建议
- [如有]
```

## 与MEMORY.md的关联

Coordinator负责维护MEMORY.md中的以下部分：
- **项目列表**：更新当前进行中的小说
- **创作数据**：累计字数和章节数
- **经验教训**：从项目管理中总结的新经验

## 注意事项

1. **不阻塞创作**：管理流程应辅助创作，不是障碍
2. **保持简洁**：避免过度文档化
3. **用户控制**：所有决策由用户最终确认
4. **灵活应变**：流程可根据实际情况调整
